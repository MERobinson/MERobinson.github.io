1.3 Downloading Public Data

You might frequently want to re-analyse published data, or compare it with your results. This section runs through how to download some example data.

We’ll be downloading the raw RNA-seq data from our labs recent publication:

First lets make a folder to store the data in, we’ll call it fastq:

mkdir fastq

1.3.1 GEO & SRA

The most commonly used database to store expression-related data is the Gene Expression Omnibus (GEO) repository. GEO archives processed data and brokers the raw sequencing reads to the Sequence Read Archive (SRA).

The format of data stored on GEO is fairly un-structured and is mainly up to the submitter, so you’ll have to read the records and/or publication to determine what is actually provided. The SRA records on the other hand should always contain raw read data.

Unaligned read data is usually stored in the FASTQ format. To extract FASTQ data from SRA we’ll be using the fastq-dump tool from the SRA-toolkit. For this we just need the SRA run (SRR) ID, of the form ‘SRR*****’. You can find this via either the SRA or GEO record for each sample, for the current example the SRR of interest are listed below.

The default behaviour of the fastq-dump tool is a little odd so we need to add some extra arguments. This makes the commands below look complicated but you don’t need to worry about all the options right now - most of them just ensure you get a standard pair of FASTQ formatted files, see the full explanation here if you’re interested.

The important parts are at the end: --outdir fastq tells the program to output the FASTQ files to the ‘fastq’ directory we just made. The last argument should be the SRR ID for the chosen run.

Copy each command below one line at a time to your terminal and run, they should take ~?? to download and extract depending on your connection.

fastq-dump --readids --dumpbase --gzip --clip --skip-technical --read-filter pass --split-files --outdir fastq  SRR
fastq-dump --readids --dumpbase --gzip --clip --skip-technical --read-filter pass --split-files --outdir fastq  SRR
fastq-dump --readids --dumpbase --gzip --clip --skip-technical --read-filter pass --split-files --outdir fastq  SRR
fastq-dump --readids --dumpbase --gzip --clip --skip-technical --read-filter pass --split-files --outdir fastq  SRR

1.3.2 The FASTQ format

Unaligned read data is usually stored in the FASTQ format, it’s text-based (i.e. human-readable) so we can take a quick look using the head command - the record for each read is (usually) composed of 4 lines, so we’ll just print the first record by specifying -n 4:

head -n 4 fastq/???

This should have printed the following lines:

@HISEQ:555:C8NT6ACXX:5:1101:1277:1979 1:N:0:TGACCA
NTCTGGTGTCAGGCCTCCTCACTGTGAGTCATCAGTACAGAGGCACAGGGAGGTGAGGTGGGAAGTAAATCGGGGTGCAGGAAATGGGAAGAGAATGGGA
+
#4=DDFDFHHHHHJJJJJJJJJJJIJJJHJIJJJIHIJJJJJJIIIHIJJGHI;CFHG@DHGHFHHFFFFFFDDDD?BCCDBBDDDDDCDDBDDCCDCD?

FASTQ (usually) contains 4 lines per read:

  • Line 1 = (unique) Read ID
  • Line 2 = The base call (A,T,C,G or N) for each position in the read
  • Line 3 = Just a “+”! (maybe the read ID again) - pretty useless, just ignore
  • Line 4 = Quality of each base call (encoded with letters/numbers/symbols)

You probably don’t need to know what all the info in the read ID is right now but there’s an explanation here if you really want to know.

The sequence quality is rather confusingly first Phred scaled then encoded by an ASCII character, you should know about base quality scores but learning the conversion to ASCII is pretty pointless. A general rule of thumb would be letters = high quality, numbers = medium, symbols = low quality.

1.3.3 Base Quality Scores

The base quality score indicates the probability of a base call being wrong for each position. The probablity is scaled with the equation below to produce the Phred quality score:

\[ Q = -10 log_{10} P \]

And if you hate maths: “ten times the number of zeros you need to wack in front of a 1 to get the probability”

Examples:

Table 1.1: Converting base quality scores.
Phred Quality Score Probability of error Base call accuracy ASCII encoding
10 0.1 (1 in 10) 90% +
20 0.01 (1 in 100) 99% 5
30 0.001 (1 in 1,000) 99.9% ?
40 0.0001 (1 in 10,000) 99.99% I

Basic take home message: higher = better.
